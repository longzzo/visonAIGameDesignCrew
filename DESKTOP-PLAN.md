## 1. 아키텍처 결정 (근거 포함)

### 데스크톱 셸: **Electron** (Tauri 대신)

| | Electron | Tauri 2 |
|---|---|---|
| 백엔드 | **Node 내장** — 현재 서버(vite 미들웨어 ~2,500줄)를 거의 그대로 이식 | Rust 코어 — 서버 전체 재작성 또는 Node 사이드카 관리 |
| 기존 기능 승계 리스크 | **최소** (같은 언어·같은 코드) | 높음 (이식 중 기능 누락 위험) |
| MCP 클라이언트 | @modelcontextprotocol/sdk (Node) 바로 사용 | 사이드카 경유 필요 |
| 바이너리 크기 | ~90MB | ~10MB |
| 메모리 | 더 씀 | 적게 씀 |

→ "기존 기능 모든 걸 다 가져간다"가 하드 요구사항이므로 **마이그레이션 리스크 최소화가 최우선**. 크기/메모리는 1인 데스크톱 도구에서 무시 가능. **Electron 확정.**

전환 전략 = 스트랭글러 패턴 (한 번에 갈아엎지 않음):
1. **v2.0**: Electron 셸이 기존 dev 서버를 품는다 (셸이 서버 자동 기동 → 창에 로드). 더블클릭 한 번에 뜨는 "진짜 프로그램", 기능 승계 100%.
2. **v2.1**: 서버 로직을 vite.config.ts에서 `server/` 모듈로 추출 → vite(개발)와 Electron main(배포)이 공유. vite 의존 제거.
3. **v2.2**: electron-builder로 설치본(.exe) 패키징 + 자동 업데이트.

### 3D 사무실: **React Three Fiber (r3f)**

- 기존 React 스택 그대로 → 상태(store)를 2D 사무실과 **공유**. 같은 이벤트(회의 소집, PM 지시, 작업 상태)가 2D에선 CSS로, 3D에선 캐릭터 이동으로 표현됨.
- Unity로 사무실을 만드는 것도 검토했으나: 앱과 별도 프로세스·통신 계층 필요, 배포 복잡. r3f는 웹·데스크톱 공용. **r3f 확정.**
- v2.0 3D 연출: 책상 배치(기획층/개발층), 캐릭터(에이전트 색상), **작업 완료 시 PM 책상으로 걸어가 보고**, **회의 소집 시 회의실 테이블로 모임**, 작업 중 타이핑 모션, 오빗 카메라.
- 이후: 픽셀아트→로우폴리 모델 교체, 감정 표현, 낮/밤 조명.

### 오케스트레이션: **OpenClaw 유지** (걷어내지 않음 — 오너 방침 2026-07-05 확정)

> **오해 정정**: 이전 문서의 "OpenClaw 대체" 표현은 틀렸다. 오너는 OpenClaw를 걷어낼 생각이 없다.
> "LangGraph·다른 오케스트레이터"는 *대체*가 아니라 **필요할 때 대안을 검토해볼 여지**로만 열어둔 것.

- **OpenClaw는 기본 런타임으로 유지한다.** 게이트웨이·세션·MCP 레지스트리·WebChat·로컬 Ollama 지원 등
  인프라를 이미 제공하며(19명 파이프라인이 그 위에서 돈다), MCP도 네이티브 지원 확인됨
  (`openclaw mcp add/configure/tools` → 런타임에 도구 주입).
- v2.1의 **다이렉트 프로바이더는 "대체"가 아니라 "보완"**이다: 개발팀처럼 *툴 루프를 한 스텝씩 보고
  제어해야 하는*(3D 사무실·DevTaskPanel 스트리밍) 작업에 한정해 쓴다. 기획팀 등 나머지는 OpenClaw 그대로.
- **LangGraph 등 다른 오케스트레이터**: 지금은 도입하지 않는다. OpenClaw로 충분하고, 손수 짠 회의 루프도
  잘 돈다. 회의 그래프가 *재개 가능한 장시간 다단계*로 정말 복잡해질 때에만 대안으로 재검토(성역 없음).

### LLM: **로컬 우선 — 클라우드는 과도기용, 궁극 목표는 로컬 전용** (오너 방침 2026-07-05 확정)

- **최종 방향 = 로컬(Ollama)로만 돌리기.** NVIDIA 같은 클라우드 모델은 **되도록 피하고**, 로컬 모델
  품질이 부족한 과도기에만 임시로 쓴다. 매일 쓰는 개인 도구가 외부 API·쿼터·비용·프라이버시에 묶이면 안 됨.
- **NVIDIA 고정 아님 — 클라우드 프로바이더 교체 가능성도 열어둔다.** 프로바이더는 추상화돼 있어야 하고
  (다이렉트 프로바이더도 NVIDIA 하드코딩 대신 provider 설정을 읽도록 일반화 대상), 다른 클라우드(예: 자체
  키의 OpenAI 호환 엔드포인트, 다른 무료 티어)로 갈아끼울 수 있게 유지한다.
- **당면 과제(로컬화 로드맵)**: ⓐ 다이렉트 프로바이더가 **함수호출 지원 로컬 모델**(예: Ollama의 qwen2.5/
  llama3.1-tools 계열, tools API)로도 개발 작업 루프를 돌 수 있게 검증 → 클라우드 없이 개발팀 동작.
  ⓑ 로컬 함수호출 품질이 관건 — 안 되면 그 부분만 한시적으로 클라우드. ⓒ vLLM은 별도 GPU 서버(24GB+)가
  생길 때만 재검토(현 8GB 노트북·Windows에선 Ollama가 정답).
- 정리: **클라우드는 "지금 품질을 메우는 임시 버팀목", 로컬이 종착지.** 모든 프로바이더 배선은 교체 가능하게.

### MCP: **Electron main에 MCP 클라이언트 허브** (v2.1)

- `mcp.json`에 stdio 서버 등록 → Electron(Node)이 spawn·연결 → 도구 목록을 앱에 노출.
- 1차 소비자는 **개발팀 에이전트**: 다이렉트 프로바이더의 함수호출 루프에 MCP 도구를 브릿지 → Unity 프로젝트 파일 읽기/쓰기, git, 빌드 실행이 가능해짐 = "개발에 실제로 관여".
- 웹 모드에서도 동작하도록 서버 모듈 쪽에 허브를 둔다 (Electron 전용 아님).
- **OpenClaw MCP와 이중 경로 공존**: OpenClaw도 `mcp add`로 같은 서버를 자기 런타임에 물릴 수 있다.
  우리 허브는 다이렉트 프로바이더용, OpenClaw는 자기 에이전트용 — 같은 mcp.json을 참조하도록 맞춰둘 것.
- **유니티 MCP 서버**(v2.3): CoplayDev/unity-mcp(47도구, MIT) 또는 AnkleBreaker/unity-mcp-server(288도구)를
  등록하면 개발팀이 실제 유니티 에디터(씬·GameObject·스크립트·테스트)를 조종. 에디터 패키지 + MCP 서버 구조.
