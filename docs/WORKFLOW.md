# WORKFLOW.md — 요청 처리 흐름 & 오케스트레이션 시나리오

이 문서는 오너가 Vision Engine에 요청을 넣었을 때 실제로 어떻게 처리되는지를 예시로 보여준다.

## 기본 흐름

```
오너 요청 → PM 접수·분해 → 서브에이전트 위임 → 수행 → PM 취합·GDD 대조 → 오너 보고
```

## 예시 시나리오 0 — 새 프로젝트 부트스트랩 (장르 미정 상태)

**오너**: "게임 하나 만들어보고 싶은데 아직 뭘 만들지 정하지도 않았어."

1. PM이 `MASTER_GDD.md`를 확인 → 1~2번 섹션이 비어 있음을 감지.
2. `templates/PROJECT_INTAKE.md`의 Stage 0 질문 2~3개만 먼저 던짐:
   - "지금 갖고 있는 게 뭔가요? (한 줄 컨셉이든, 그냥 막연한 상태든 상관없어요)"
   - "장르에 대한 감이 있나요?"
   - "플랫폼은 모바일/PC/콘솔/미정 중 어디에 가까운가요?"
3. 오너 답변에 따라 즉시 GDD 1번 섹션 일부를 채운다.
4. 장르가 여전히 안 나오면 Stage 1(핵심 판타지·톤)로 계속 진행 — 장르 중립적 질문이라 막힘없이 이어감.
5. 장르가 드러나면 해당 장르의 "장르별 추가 질문"으로 전환.
6. 핵심 항목이 채워지면 부트스트랩 종료 → 통상 위임 절차로 전환. 오너에게 "아직 안 정한 부분"을 요약 보고.

> 이 시나리오가 다른 시나리오들과 다른 점: 서브에이전트에게 위임하기 전에 PM과 오너 사이의 인테이크 대화가 먼저 일어난다.

## 예시 시나리오 1 — 복합 요청

**오너**: "5번째 스테이지에 등장하는 보스전을 기획해줘. 스토리, 전투 방식, 밸런스까지."

1. **PM 분해**:
   - scenario → 보스의 배경·서사적 위상
   - systems → 전투 방식·스킬 구조
   - balance → 실제 수치(체력, 데미지, 페이즈 전환 시점)
   - (gameplay → 이 보스전이 코어 루프에 어떻게 얹히는지 확인)

2. **위임**: PM이 각 에이전트에 TASK_BRIEF 발행. 참조 GDD 섹션 명시(세계관, 코어 게임플레이, 핵심 공식).

3. **수행 순서**: scenario·systems가 먼저(구조), balance가 그 위에 수치(의존성 있음).

4. **PM 취합·검증**:
   - 보스 서사가 세계관과 맞는가? (GDD 2번)
   - 전투 구조가 기존 시스템과 정합하는가? (GDD 5번)
   - 수치가 5스테이지 난이도 곡선에 맞는가? (GDD 6번)
   - 세션 길이 가정(예: 보스전 5분) 안에 들어오는가?

5. **오너 보고**: 통합된 보스전 기획서 + GDD 변경 제안(신규 보스를 고유명사 사전에 추가).

## 예시 시나리오 2 — 단일 에이전트 요청

**오너**: "시작 화면 UI 구조 잡아줘."

1. PM → uiux 단독 위임 (참조: GDD 세계관·플랫폼).
2. uiux가 화면 목록·여정·와이어프레임 작성.
3. PM이 톤 정합성만 확인 후 오너에게 전달.

> 단순 요청은 굳이 여러 에이전트를 거치지 않는다. PM이 판단해 최소 경로로 처리한다.

## 예시 시나리오 3 — 모순 발견

**오너**: "이 신규 무기를 게임 최강으로 만들어줘."

1. balance가 수치 설계 중 발견: 최강 수치가 기존 난이도 곡선을 붕괴시킴.
2. balance는 임의로 진행하지 않고 "오픈 이슈"로 표시해 PM에 반환.
3. PM이 트레이드오프를 정리해 오너에게 선택지 제시:
   - (A) 최강으로 하되 획득을 극도로 제한
   - (B) 특정 상황에서만 강한 조건부 설계
   - (C) 밸런스 범위 안에서 "강한 편"으로 조정
4. 오너 결정 → 반영 → GDD 갱신.

## GDD 쓰기 규칙 (동시성 보호)

- 서브에이전트는 GDD를 **읽기만** 한다.
- GDD 수정은 PM이, 오너 승인 후, 단독으로 수행한다.
- 이유: 여러 에이전트가 동시에 GDD를 고치면 충돌·불일치가 생긴다. 단일 쓰기 지점으로 일관성을 보장한다.

## 확장 단계별 가동 범위

| 단계 | 가동 에이전트 | 목적 |
|---|---|---|
| 1 | PM + scenario | 최소 파이프라인 검증 |
| 2 | + systems, balance | 전투·밸런싱 루프 확인 |
| 3 | + gameplay, uiux, bm | 전체 7역할 가동 |
| 4 | (프론트엔드 추가) | 응답 시각화 |
| 5 | (보조 무료 API 연결) | 대량 작업 오프로드 |
